NetGauge: Testing Methodology

Team Ookla - November 11, 2016 23:00

TCP Test Components

Latency/Jitter

  1. This test is performed by measuring the time it takes for the server to reply to a request from the Client. The client sends a message to the server, upon receiving that message, the server sends a reply back. The round-trip time is measured.
  2. This test is repeated multiple times (configurable) with the lowest value determining the final result.
  3. The variance for all of the samples determines jitter.

Download

  1. The client establishes multiple connections with the server over the defined port based on the initial threads parameter. The client requests the server to send a configurable initial chunk of data (configurable).
  2. The client calculates the real-time speed of the transfers, then adjusts the chunk size and buffer size based on this calculation to maximize usage of the network connection.
  3. As the chunks are received by the client, the client will request more chunks throughout the duration of the test.
  4. During the first half of the test, the client will establish extra connections to the server if it determines additional threads are required to more accurately measure the download speed.
  5. The test ends once the configured amount of time has been reached.

Upload

  1. The client establishes multiple connections with the server over the defined port and sends a configurable initial chunk of data (configurable).
  2. The client calculates the real-time speed of the transfers and adjusts the chunk size and buffer size based on it to maximize usage of the network connection, and requests more data.
  3. As the chunks are received by the server, the client will send more chunks throughout the duration of the test.
  4. During the first half of the test, the client will establish extra connections to the server if it determines additional threads are required to more accurately measure the upload speed.
  5. The test ends once the configured amount of time has been reached.

 

Java Test Components

Packet Loss

  1. The client sends multiple UDP packets to the server over the defined port, taking a small pause in between packets
  2. The server receives those packets, and informs the client on how many were received.
  3. The client calculates the packet loss by comparing the number of packets sent vs received.

Firewall Test

  1. The client attempts to connect to the server via TCP over the defined port. (TCP to Server test)
  2. The client sends a UDP packet to the server (UDP to Server test)
  3. The client requests the server to attempt a TCP connection to the client (TCP to Client test)
  4. The client requests the server to send a UDP packet to the client (UDP to Client test)

 

HTTP Legacy Fallback Testing (Optional Configuration)

Latency/Jittter

  1. This test is performed by measuring the time it takes to get a response for a HTTP request sent to the web server
  2. This test is repeated multiple times (configurable) with the lowest value determining the final result.
  3. The variance for all of the samples determines jitter

Download

  1. Small binary files are downloaded from the web server to the client to estimate the connection speed
  2. Based on this result, one of several file sizes is selected to use for the real download test
  3. The test is performed with cache prevention via random strings appended to each download
  4. Up to 16 parallel HTTP threads (configurable) can be used for the test
  5. Throughput samples are received at up to 30 times per second
  6. These samples are then aggregated into 20 slices (each being 5% of the samples)
  7. The fastest 10% and slowest 30% of the slices are then discarded (see * below for more detail)
  8. The remaining slices are averaged together to determine the final result

Upload

  1. A small amount of random data is generated in the client and sent to the web server to estimate the connection speed
  2. Based on this result, an appropriately sized set of randomly generated data is selected for upload
  3. The upload test is then performed in configurable chunk sizes (pushed to a server-side script via a POST)
  4. The test can be done using up to 16 parallel HTTP threads (configurable)
  5. Chunks are sorted by speed, and the fastest half is averaged to eliminate anomalies and determine the result

Note: When using HTTP Legacy Fallback, the end of test panel will not provide an upload graph due to the difference in testing methodology.
