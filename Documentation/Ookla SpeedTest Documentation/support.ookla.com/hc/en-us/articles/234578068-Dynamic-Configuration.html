Dynamic Configuration

Team Ookla - November 11, 2016 19:21

Ookla NetGauge has a flexible configuration system that can dynamically alter its settings based on specific factors. Currently, this feature can be used to dynamically control integer (natural numbers) and boolean (true/false) values.

Using dynamic values

To assign a dynamic value to a specific parameter, give the parameter a value that begins with “dyn:” followed by the name of the condition-set that will be used.

<socket-upload initialthreads="dyn:tcpulthreads" maxthreads="32"/>

The above example indicates that the number threads initially used for the upload test would be determined by the 'tcpupload' dynamiv value.

Configuring conditions for dynamic values

The condition sets are parsed in order, and all attributes of the condition must be true for its value to be used. For each unique conidtion name in the condition set, the value of the first condition where all attributes are true is assigned to the condition variable. A condition that only contains a name and value attribute is considered true and its value will be assigned to the condition variable.

<conditions>
<cond name="tcpulthreads" download="+40000" value="8"/>
<cond name="tcpulthreads" download="+10000" value="4"/>
<cond name="tcpulthreads" value="2"/>
</conditions>

The example above contains the following logic:

  • If the download test result is greater than 40 Mbps, then start the upload test with 8 threads
  • If the download test result is greater than 10 Mbps, then start the upload test with 4 threads
  • Otherwise, start the upload test with 2 threads

Conditional operators allowed in a condition

Operatormeaningexampleexample meaning
+ greater than (>) latency=”+20” latency is higher than 20ms
- less than(<) download=”-5000” download is lower than 5000kbps
= equals clientip=”=192.168.1.1” IP is 192.168.1.1
/ range latency=”/50-100” latency is between 50ms and 100ms

Available evaluation values

  • latency
  • download
  • upload
  • serverid
  • clientip

When building your own dynamic configuration rules, note that you can only use elements as they become available. For example, a dynamic value in the download test could not depend on the upload test value as a condition, since the upload test takes place chronologically after the download test.

Popular Recipes

Use more threads with internal users

<socket-download initialthreads="dyn:threads" />
<socket-upload initialthreads="dyn:threads" />
<conditions>
<!-- if the user has an internal IP address, start with more threads -->
<cond name="threads" clientip="/192.0.0.0-192.255.255.255" value="32" />
<!-- default condition, start with 8 threads -->
<cond name="threads" value="8">
</conditions>

Enable throttling on a specific server only

<throttling testlimit="dyn:throttle" />
<conditions>
<!-- if the user is testing against server 2, enable throttling -->
<cond name="throttle" serverid="2" value="10" />
<!-- default condition, disable throttling for all other users -->
<cond name="throttle" value="0">
</conditions>
