Fix for latency error after upgrading to flash 24.0.0.186

Brian Turley - December 28, 2016 23:52

We investigated the recently reported latency error issue and found that this is an issue with Adobe Flash 24.0.0.186. Adobe is aware of the issue and has already released a beta to address this. Hopefully the release version will be out soon.

https://tracker.adobe.com/#/view/FP-4198184

Until the next version of Flash is released, NetGauge customers and hosts that want to continue using TCP for web tests have two options. You can either set your firewall to actively refuse connections on port 843, or configure a Flash policy daemon to respond on 843.

It seems that most firewall configurations don't actively refuse the connection on port 843, but instead just block it without actively responding with a connection refused message. This causes Flash to wait too long before timing out and falling back to the normal (pre-update 24) behavior. If the connection is actively refused, Flash immediately falls back to requesting the policy file on the port used for testing.

The second option is to set up a Flash policy daemon on all of the Ookla Servers. It's just a script that listens on port 843 and returns the contents of an XML file.

Detailed instructions can be found at http://www.adobe.com/devnet/flashplayer/articles/socket_policy_files.html.

The only edit that needs to happen is in the included flashpolicy.xml file. Delete the contents of the file and change it to:

<cross-domain-policy>

<allow-access-from domain="*" to-ports="8080"/>

</cross-domain-policy>

If you don't use 8080 (which is the default), just change that to the port you have configured.

Each Ookla Server needs to either actively refuse connections on port 843 or run the Flash policy daemon.

Please try these options and submit a support ticket if you have any additional questions.
