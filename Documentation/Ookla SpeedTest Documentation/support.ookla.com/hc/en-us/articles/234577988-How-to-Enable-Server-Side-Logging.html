How to Enable Server-Side Logging

Brian Turley - January 05, 2018 19:58

Enabling server-side logging for the NetGauge OoklaServer Daemon is easy. First, you will need to edit the file OoklaServer.properties. You can do so by opening it in a text editor and uncommenting the five lines below the line # Log to files. This is done by removing the hash (#) from the beginning of each line. Once done, you will need to comment out the four lines under # Log to the console by adding a hash to the beginning of each line. Please see the KB article Ookla Server Daemon Advanced Configuration for more information about other available server daemon settings.

Where are the logs stored?

By default, the log file is stored in the same directory as the server daemon in a file named ooklaserver.log. The location and filename can be changed by editing the OoklaServer.properties file. Edit the value of logging.loggers.app.channel.path to equal the desired folder and file name. For example, the default entry is:

logging.loggers.app.channel.path = ${application.dir}/ooklaserver.log

 

If you would like to change the name of the log file and store it in a separate directory, /System/Library/Logs/ in this example,  you could modify the line to equal:

logging.loggers.app.channel.path = /System/Library/Logs/speedtest.log

 

What kind of information is stored in the log file?

This depends on the logging level specified in OoklaServer.properties. Two logging levels are available, Information and Error. The Error level logging will only store information about the cause of encountered errors. For example, if a user navigates away from the page midway through the download test, the following error may be recorded:

2013-10-10 00:39:12 [17633 - 18251] [Error] Disconnect IOException: I/O error: 32

 

By default, the logging level is set to information. To enable error only logging, change the value of logging.loggers.app.level from information to error. If the logging level is left set to information, all server daemon activity is logged. This can cause lengthy log files. Here is a snippet from an actual log file showing only the information generated the first second of a new client connection:

What is the approximate file size after a day of logging?

This also depends on a few factors. First of all, if your logging level is set to only capture errors, your file size will be extremely small, even with a large number of users.

If your logging level is set to information, the file size can still vary depending on the client's connection speed. A single test where the client has a bi-directional connection of approximately 1.5 Mbps, results in a log file about 32kb in size, while a single test from a client with a bi-directional gigabit connections results in a log file anywhere from 700 KB to 1.5 MB, depending on the client configuration template specified in the NetGauge settings.
