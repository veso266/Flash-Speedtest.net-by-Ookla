NetGauge: What's New?

Team Ookla - November 11, 2016 22:23

New Testing Method

Ookla has long been considered the standard for measuring Internet and Wireless connection speeds, in fact, over the past five years, our industry-leading testing engine (v2.0) has "set the pace" for thousands of companies, and powered over four billion tests around the world! More recently, our ISP and Mobile Carrier Clients as well as the U.S. FCC, have pushed this technology to its limits. Because of this, we have been hard at work for the past two years creating our next generation testing platform. This Article is intended to explain the differences between these two testing engines and give an overview of the unique features and benefits of our new V3 (Software Version 3.0) offerings.

Our legacy v2.0 applications such as Speedtest and Line Quality as well as specialized tests such our as VoIP Test, all measured MST ("maximum sustainable throughput") via HTTP, transferring compressed data over one or more data streams ("threads") to and from a properly configured Speedtest (web) server. Meanwhile as networking has evolved, Ookla is determined to keep pace. We have done that and much more, as we set out to begin creating our v3.0 engine over two years ago. Clients can now use our technology to test:

  • Very fast connections: FTTH/FTTB (Fiber to the Home or Business) 
  • LAN testing: Including: Gigabit Ethernet, and even faster speeds 
  • Inherently high latency connections: 3G/4G/Wimax

Responding to the requirements of above, Ookla completed development of our V3 engine, written from the ground up, our latest core testing modules. Today we offer this cutting-edge technology primarily via our licensed NetGauge software. NetGauge licensing is available only from Ookla. We also offer the ability to test for free through our flagship site, Speedtest.net. 

The V3 engine works at the transport layer (TCP) instead of the application layer (HTTP). This brings us closer to the Internet connection, and this means we can better control it. This also allows us to bypass most client side proxies, which is much more difficult to accomplish when testing over HTTP.

Instead of depending on a passive server-side component via a web server in V2, we developed a standalone testing daemon that will receive client connections and take part in the testing process. When testing over TCP in NetGauge, we can control the connection from both sides. This means we can ensure that the conditions and the settings are ideal. This is accomplished via a bidirectional exchange of information between the client and server which dynamically determines the best parameters for the test as it runs. This makes NetGauge the most accurate engine created by Ookla to date.

Legacy Fallback Support

NetGauge tests are no longer performed against a webserver via HTTP, instead it uses an alternative TCP port (TCP 8080 by default). This can potentially cause issues with users performing tests from behind a restricted network (firewalled connections or corporate networks). To ensure that NetGauge could always provide a test result, we incorporated an optional HTTP Legacy Fallback system that when configured, will try different methods or servers to complete the test.

This is useful in the following cases:

  • If port X is blocked, the client will try on port Y instead and complete the test.
  • If server X is down, the client will try server Y instead and completes the test.
  • If servers X and Y are both unreachable via all defined V3 TCP ports (blocked), the client will legacy fallback and perform a V2-method HTTP test. This requires configuration and a standard web server with a supported scripting language (ASP, ASPX, PHP, JSP).
