Why do I get better speeds when I go to Speedtest.net?

Team Ookla - November 11, 2016 19:21

To overcome throughput limits caused by response times and the TCP window, the Ookla's NetGauge engine supports testing parallel threads over multiple TCP sockets. Speedtest.net host locations support NetGauge's default socket testing methodology. When a socket connection can not be established, our host locations support HTTP Legacy Fallback, which uses up to four HTTP threads to better saturate higher-speed connections.

You can find out how to enable HTTP Legacy Fallback in NetGauge in our KB: "NetGauge: HTTP Legacy Fallback"

Conflicting results between Speedtest.net and your instance of NetGauge can also be caused by differences in the configuration templates used. Adjusting the configuration template being used is highly recommended if you are testing higher capacity circuits.

If you continue to have trouble reaching the speeds recorded on Speedtest.net, make sure your test server has enough throughput to accurately measure the connection speeds you're testing. If your server is being overwhelmed by too many simultaneous tests during peak periods, ask your account executive about our throttling add-on, which can limit the number of concurrent tests.

If your server is active on Speedtest.net and you see better speeds when you test to an off-network server, or you see better speeds from the Speedtest.net user experience, please take note of the current configuration used on Speedtest.net:

 

HTTP Legacy Fallback Settings

<download testlength="10" initialtest="250K" mintestsize="250K" threadsperurl="4"/>

<upload testlength="10" ratio="5" initialtest="0" mintestsize="32K" threads="2" maxchunksize="512K" maxchunkcount="50" threadsperurl="4"/>

<latency testlength="10" waittime="50" timeout="20"/>

 

NetGauge Socket Testing Settings

<socket-download testlength="15" initialthreads="4" minthreads="4" maxthreads="32" threadratio="750K" maxsamplesize="5000000" minsamplesize="32000" startsamplesize="1000000" startbuffersize="1" bufferlength="5000" packetlength="1000" readbuffer="65536"/>

<socket-upload testlength="15" initialthreads="dyn:tcpulthreads" minthreads="dyn:tcpulthreads" maxthreads="32" threadratio="750K" maxsamplesize="1000000" minsamplesize="32000" startsamplesize="100000" startbuffersize="2" bufferlength="1000" packetlength="1000" disabled="false"/>

<socket-latency testlength="10" waittime="50" timeout="20"/>

 

Dynamic Configuration (Note the variables in bold within the socket-upload settings)

<conditions>

<cond name="tcpulthreads" download="+100000" value="8"/>

<cond name="tcpulthreads" download="+10000" value="4"/>

<cond name="tcpulthreads" value="2"/>

</conditions>
