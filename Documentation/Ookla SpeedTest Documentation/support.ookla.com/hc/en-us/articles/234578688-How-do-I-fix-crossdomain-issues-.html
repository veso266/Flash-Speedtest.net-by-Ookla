How do I fix crossdomain issues?

Michael Jackson - May 24, 2017 22:22

This article will assist with troubleshooting crossdomain issues.

What is the crossdomain.xml file?

The crossdomain.xml file is a cross-domain policy file. It grants the Flash Player permission to talk to servers other than the one it's hosted on and is required for Flash to use Speedtest servers.

Note there are two sources of crossdomain information for a Speedtest Server.

For HTTP Legacy Fallback the crossdomain.xml file at the root of your server is used. For example: http://speedtest.mycompany.com/crossdomain.xml

For OoklaServer the OoklaServer.properties setting OoklaServer.allowedDomains determines how an internal crossdomain file is generated.

In what circumstances do I need to use one?

You need a crossdomain.xml file when your Speedtest Server uses external hosts as testing servers.

  • You are, or are applying to be a Speedtest Server sponsor.
  • The Speedtest application is hosted on your main webserver (www.mycompany.com for example), but you have a dedicated testing server that the test uses. (speedtest.mycompany.com)
  • You have multiple servers

What should it look like?

HTTP Legacy:

If your company's website is hosted at www.mycompany.com and you also wanted to allow speedtest.net to use your server, your crossdomain would look like:

<?xml version="1.0"?>
<cross-domain-policy>
    <allow-access-from domain="*.ookla.com" />
    <allow-access-from domain="*.speedtest.net" />
</cross-domain-policy>

This file goes in the root of document folder on the server you want to allow access. In the example above, the crossdomain.xml file would need to be located at http://speedtest.mycompany.com/crossdomain.xml.

OoklaServer:

In the same directory you are running OoklaServer from you will have an OoklaServer.properties file. Within this file, the OoklaServer.allowedDomains setting controls how an internal crossdomain file is generated.

By default this line is commented out and any Flash applet will be able to use your server. To restrict access, you can uncomment this line and leave *.ookla.com, *.speedtest.net which will allow speedtest.net to use your server. You can also add *.mycompany.com to the list to include access to your domains.

Example allowedDomains line:

# OoklaServer.allowedDomains = *.ookla.com, *.speedtest.net

Note: There was a bug in previous OoklaServer.properties files which included ookla.com, speedtest.net without the *. in front of the domains. This is incorrect and will break OoklaServer functionality when testing with Flash and result in falling back to HTTP Legacy. Ensure the *. precedes any included domains.

Why am I getting a notice/error in the Server Tester about crossdomain?

First you need to determine if you are seeing an issue with crossdomain.xml which is part of HTTP Legacy or crossdomain which is part of OoklaServer. Note that there are two crossdomain tests in the Server Tester and its important to consider if the test that failed is for HTTP Legacy or OoklaServer.

HTTP Legacy crossdomain.xml

The easiest way to check your HTTP Legacy crossdomain.xml file is to go to your HTTP Legacy server upload URL and replace the file path with "crossdomain.xml"

For example, if your server upload URL looks like this:

http://speedtest.mycompany.com/speedtest/upload.php

You should drop the path and replace it with crossdomain.xml to get this:

http://speedtest.mycompany.com/crossdomain.xml

This file should load properly and contain the text above from the "What should it look like?" section in this document.

OoklaServer crossdomain

You may see a notice entry for your OoklaServer crossdomain test indicating that "The detected cross domain policy allows all domains to test to this server."

This is due to the default setting for allowedDomains within your OoklaServer.properties file being * which allows all sites. Note that this will not prevent your server from working, it just means that any Flash applet will be able to use your server for testing.

To correct this, uncomment the OoklaServer.allowedDomains setting within your OoklaServer.properties file. Review the "What should it look like?" section above for OoklaServer for more information.
