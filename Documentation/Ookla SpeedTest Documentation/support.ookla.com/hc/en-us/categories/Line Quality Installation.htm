Line Quality Installation

Team Ookla - November 11, 2016 21:21

 Line Quality Test Installation Instructions

To setup the Line Quality Test files on your server, you will first need to have a web-accessible folder. On many Linux distributions, /var/www/ is such a folder. For most Windows servers this will be C:\Inetpub\wwwroot\.

You will also need to have Java (either Sun's JRE or OpenJDK) functioning properly on the server. You can verify it is working via the command-line by typing this:

java -version

If successful then you will see output similar to this:

[root@server]$ java -version
java version "1.5.0_14"
Java(TM) 2 Runtime Environment, Standard Edition (build 1.5.0_14-b03)
Java HotSpot(TM) 64-Bit Server VM (build 1.5.0_14-b03, mixed mode)

Package Contents

Unzipping the contents of the provided Zip archive will reveal two folders named client and server. Both sets of files actually go on the server, but the client ones are delivered to the client browser via a web server. Place the client contents somewhere inside the web-accessible folder on your server keeping the directory structure intact. The hierarchy and contents should then look like this:

 crossdomain.xml Flash security file specifying host and domain access
 debug.html debug version of sample HTML page for the Line Quality Test
 deployJava.js deployJava Javascript library for properly loading Java applications
 expressInstall.swf support file for SWFObject to handle Flash installations
 functions.js Javascript called from Flash after each Line Quality Test completes
 index.html sample commented HTML page for the Line Quality Test
 linequalitytest.swf core Line Quality Test Flash application
 LQApplet.jar unsigned core Line Quality Test Java client application
 settings.* settings files for the Speed Test includes license key
 swfobject.js SWFObject Javascript library for properly loading Flash applications
 swfobject.js SWFObject Javascript library for properly loading Flash applications
 VoipApplet.jar signed core Line Quality Test Java client application
 speedtest/crossdomain.xml duplicate Flash security file specifying host and domain access
 speedtest/random*.jpg highly randomized sample files used for download test
 speedtest/upload.* scripts used for the upload test
The contents of the server folder are:
 pv.exe  Windows command line process viewer 
 pv.txt  Windows command line process viewer documentation and disclaimer
 voipcheck.bat  batch file for starting up Line Quality Test daemon on Windows
 voipcheck.sh  shell script for starting up Line Quality Test daemon on UNIX/Linux
 VoipServer.jar  core Line Quality Test Java server daemon application
These files should now be accessible online at the host you setup when you configured your web server. For this example, we will assume that you've setup lqtest.mydomain.com. In your browser, go to http://lqtest.mydomain.com. You should see your Line Quality Test! It won't work completely yet though.

Java Daemon Installation

Place the contents of the server folder somewhere else on your server. It is recommended that you choose a location that is not web-accessible. For UNIX/Linux we recommend creating a new user called ookla and then placing the files into that home directory:

/home/ookla/

For Windows we recommend creating this directory to place them into:

C:\Program Files\Ookla\

Open a command prompt on the server and change to the directory containing VoipServer.jar. Then type the following to make sure daemon starts up properly:

[ookla@server]$ java -jar VoipServer.jar
LOG Thu, 09 Aug 2007 22:10:07 -0700 MAIN No IP Specified, will use: 69.17.117.196
LOG Thu, 09 Aug 2007 22:10:07 -0700 MAIN Start UDP Thread
LOG Thu, 09 Aug 2007 22:10:07 -0700 MAIN Start TCP Listener
LOG Thu, 09 Aug 2007 22:10:07 -0700 S-TCP Listening on port: 5060

Open another command prompt and try to telnet to port 5060 on your server (note that QUIT was typed manually in this session):

telnet> open demo.ookla.com 5060
Trying 69.17.117.196...
Connected to dev.ookla.com.
Escape character is '^]'.
QUIT
BYE
Connection closed by foreign host.

If your session looks the same then the Java daemon is setup correctly. Go ahead and kill the Java daemon (CTRL-C). We recommend using the provided scripts to make the daemon start up and stay running on its own. The voipcheck.sh and voipcheck.bat files serve this purpose.

On Linux you can setup voipcheck.sh to run as often as every minute via cron:

http://en.wikipedia.org/wiki/Cron

On Windows you can setup voipcheck.bat as a Scheduled Task (found in Control Panel) to run every minute by editing the advanced schedule after creating the job.

After setting either of these up, wait a minute and then perform the telnet test again to confirm things are working as expected.

Upload Test Verification

Confirm that the upload testing component is setup correctly by visiting (replace http://lqtest.mydomain.com with your host and domain):

  • If using PHP: http://lqtest.mydomain.com/speedtest/upload.php
  • If using JSP: http://lqtest.mydomain.com/speedtest/upload.jsp
  • If using ASP.NET: http://lqtest.mydomain.com/speedtest/upload.aspx
  • If using ASP: http://lqtest.mydomain.com/speedtest/upload.asp


You should see output that looks like this:

size=475

The number shown will vary. This indicates that things are setup correctly. Keep this URL at hand, as you will need it to finish the configuration (see Setting the Server URL and Host below). If you see the source code of the script or an error of any kind, then something is not setup right with the chosen scripting language on your web server.

Setting the Server URL and Host

  1. Open the settings.* file on your web server in a text editor (on Windows be sure to use Notepad and not Wordpad).
  2. Look for the line that starts with <server url=. It should look something like this:
    <server url="http://lqtest.mydomain.com/speedtest/upload.php" linequality="lqtest.mydomain.com" />
  3. Please be sure to update both the url= and linequality= sections.
